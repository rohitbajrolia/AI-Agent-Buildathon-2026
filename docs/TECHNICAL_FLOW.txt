Home Insurance Coverage Review - Technical Flow (Developer Guide)
=================================================================

Read this file start-to-finish once to understand the application end to end. After that, you should be able to:
- run the system end-to-end
- understand how indexing + retrieval + answering work
- find the exact file that implements each feature
- extend the MCP tools or the LangGraph workflow safely


0) What this application is (and what it is not)
------------------------------------------------

What it is:
- A local-first homeowners insurance Q&A app that answers using only your policy packet (PDFs/images) indexed into Qdrant.
- It is designed to support auditability: it shows sources, keeps a trace, and can fail closed (block answers) when it cannot defend citations.

What it is not:
- Not legal advice, underwriting, or a binding coverage decision. The UI shows this disclaimer at the top.
- Not a general-purpose insurance question-answering system. The prompt rules and validation gates block unsupported answers.


1) Repository layout
--------------------

Top-level:
- docker-compose.yml
  Starts Qdrant with a persistent Docker volume.

- run_demo.sh / run_server.sh / run_ui.sh / stop_demo.sh
  Helper scripts for starting and stopping the demo locally (Windows + Git Bash).

Server package:
- home-insurance-mcp/
  - src/server/main.py
    Entry point that calls server.mcp_server.main().
  - src/server/mcp_server.py
    The MCP tool server. Owns ingest/index/retrieve + readiness checks.

Client package:
- home-insurance-mcp-client/
  - src/client/app.py
    Streamlit UI (buttons, guardrails, panels, downloads).
  - src/client/graph.py
    LangGraph state machine (plan -> retrieve -> precedence_check -> validate -> answer -> citation_verify).
  - src/client/mcp_client.py
    Async MCP client wrapper (Streamable HTTP calls to the server tools).
  - src/client/prompts.py
    System prompt + output format rules (including citation formatting rules).
  - src/client/ui_theme.css
    Optional UI styling loaded by app.py if present.

Local-only state (git-ignored):
- docs/
  Default docs root for policy PDFs. The server enforces that ingest/index folder paths stay under this root.

- .demo_state/
  Small local files written by the UI/server:
  - docs_fingerprint.json (client): used to detect "docs changed" and gate re-indexing
  - feedback.jsonl (client): optional "Helpful / Needs work" clicks from the UI
  - handoff_tickets.jsonl (server): persisted handoff tickets when local disk writes succeed


2) Technology stack
-------------------

Runtime / framework:
- Python 3.12+
- Streamlit (client UI)
  Used for fast UI: sidebar controls, progress panels, audit log, and downloads.

- LangGraph (client workflow)
  Used to make the agent behavior explicit and inspectable as a state machine with gates.

Tool protocol:
- MCP (Model Context Protocol) over Streamable HTTP
  The client calls server tools as typed "tool calls" (health, index_status, ingest/index/retrieve, etc.).

Vector store:
- Qdrant (Docker)
  Stores embeddings for the indexed policy packet. Retrieval uses semantic search.

Model provider:
- OpenAI Python SDK
  - server: embeddings (`text-embedding-3-small`) for indexing and query embedding
  - client: chat completions (`gpt-4.1-mini`) for drafting an answer from retrieved SOURCES

Document parsing / OCR:
- pypdf (server): PDF text extraction
- PyMuPDF/fitz + Pillow + pytesseract (server): OCR fallback for scanned PDFs and images
  OCR fallback is gated by env (`ENABLE_PDF_OCR_FALLBACK`) and runtime availability.

Config:
- python-dotenv
  Both apps load `.env` files locally with override enabled.


3) How you run it (start here)
------------------------------

Step A - Start Qdrant
- docker compose up -d

Step B - Start MCP server + Streamlit UI
- easiest: bash ./run_demo.sh
  This script:
  - checks Qdrant health first
  - checks MCP health via an MCP tool call (not a plain GET)
  - starts the MCP server if needed
  - starts Streamlit UI

Manual start:
- server: run `home-insurance-mcp/src/server/main.py` (it starts Uvicorn/Starlette)
- client: run Streamlit on `home-insurance-mcp-client/src/client/app.py`


4) End-to-end technical flow (from click to answer)
---------------------------------------------------

This system is intentionally split into: "tools" on the server, and "agent workflow + UI" on the client.

4.1 UI startup and readiness

File: home-insurance-mcp-client/src/client/app.py

On load:
- Streamlit config + optional CSS theme.
- Shows an always-on disclaimer and a PII warning expander.
- Reads/updates Streamlit session state for:
  - docs directory input
  - index status payload
  - consent checkbox
  - require citations toggle
  - last run outputs (answer, sources, trace, etc.)

Readiness signals:
- The UI calls server tool `index_status` and uses it to decide if the demo is "ready":
  - Qdrant reachable
  - collection exists and has points
  - OpenAI is configured (and optionally ping-checked)
  - OCR readiness signals (fitz + tesseract) are shown for scanned packets


4.2 Indexing flow (ingest -> index into Qdrant)

Server tools involved:
- ingest_folder (summary only)
- index_folder_qdrant (chunk + embed + upsert)
- start_ingest_job / start_index_job + job_status (same work, but with progress polling)

Client UI entry points:
- "Ingest Folder" button
- "Index to Qdrant" button
- Optional: "Self-check" panel runs health -> index_status -> sample retrieve

File: home-insurance-mcp-client/src/client/app.py
What the UI does:
- Reads docs folder path from the sidebar.
- Uses job tools so the UI can show progress:
  - start_ingest_job / start_index_job return a job_id
  - UI polls job_status and updates the progress display
- Re-index gating:
  - If the Qdrant collection already exists, indexing is disabled unless the user explicitly confirms docs changed.
  - The client computes a document fingerprint from PDF path + size + mtime and stores it in `.demo_state/docs_fingerprint.json`.

File: home-insurance-mcp/src/server/mcp_server.py
What the server does during indexing:
- Enforces docs root:
  - DOCS_ROOT defaults to `<workspace>/docs` (override via `MCP_DOCS_ROOT`)
  - ingest/index folder paths must be under DOCS_ROOT

- Extracts text:
  - PDFs: pypdf extracts page text
  - OCR fallback for scanned pages:
    - if extraction is empty, and OCR is enabled, it renders the page with PyMuPDF (fitz) and OCRs with pytesseract
  - Images: OCR directly via pytesseract

- Classifies doc type (heuristic):
  - policy_booklet / declarations / endorsement / unknown
  Used later as metadata and to filter retrieval.

- Chunks text (default settings used by both ingest and index):
  - chunk_size: 1200 chars
  - overlap: 150 chars

- Embeds chunks:
  - OpenAI embeddings model: text-embedding-3-small

- Upserts to Qdrant:
  - Collection name: `QDRANT_COLLECTION` (default: home_insurance_docs)
  - Payload stored per chunk includes:
    - file_name (path relative to the indexed folder)
    - doc_type
    - page_number (PDFs)
    - chunk_index (stable index used in citations)
    - text (full chunk)

Point IDs are stable UUIDv5 based on a deterministic string:
- PDFs: "<rel_path>::p<page>::c<chunk_in_page>"
- Images: "<rel_path>::<chunk_index>"


4.3 Question answering flow (LangGraph + MCP tools + gating)

Entry point:
- UI button "Review Coverage" in home-insurance-mcp-client/src/client/app.py
  This builds a GraphState and calls graph.invoke().

Workflow file:
- home-insurance-mcp-client/src/client/graph.py

GraphState (the data that flows through the nodes):
- question (required)
- state (jurisdiction context, default "IL")
- require_citations (default True)
- retrieval_plan (redacted previews)
- raw_results (retrieved matches with metadata)
- sources (the SOURCES block shown in the UI and passed to the model)
- answer (drafted response)
- validation + blocked (fail-closed gates)
- endorsement_signals + precedence_check (override risk signals)
- trace (audit events)

Node 1: plan
- Function: plan_retrievals_node()
- Builds a small retrieval plan (up to ~5 targeted sub-queries):
  - Coverage grant
  - Definitions
  - Exclusions / limits
  - Endorsements / changes
  - Conditions / duties (only for certain questions)
- Stores a redacted version of the plan in state["retrieval_plan"] and logs it to trace.

Node 2: retrieve (multi-retrieve)
- Function: multi_retrieve_node()
- For each plan query:
  - Calls server tool retrieve_clauses (via client.mcp_client.retrieve_clauses)
  - Collects and merges results across topics
- De-dupes results by a stable key:
  (file_name, doc_type, chunk_index, page_number)
- Keeps top matches and builds the SOURCES block:
  - Each line includes a citation tag plus a short snippet preview.
  - Citation tags look like:
    - [file | doc_type | chunk N]
    - [file | doc_type | p. X | chunk N]

Node 3: precedence_check
- Function: precedence_check_node()
- Detects simple, defensible endorsement override risk signals:
  - If retrieved matches include doc_type == "endorsement"
  - And the endorsement snippet includes phrases like "changes the policy", "replaces", "supersedes", etc.
- Stores:
  - endorsement_signals (details + matched patterns)
  - precedence_check (a small summary with override_risk boolean)
- The UI renders this in the "Endorsement override check" expander.

Node 4: validate (evidence strength gate)
- Function: validate_node()
- Only strict when require_citations is True.
- Blocks early when evidence is clearly too weak to be safe, for example:
  - no retrieved matches
  - extremely low top relevance score
  - weak relevance + low diversity (all evidence from one narrow slice)
- Writes a structured validation object:
  - reasons, warnings, next_actions, and basic stats
- This drives:
  - the "Evidence strength" label in the UI
  - the blocked state (with next steps)

Node 5: answer (model draft)
- Function: answer_node()
- Uses:
  - home-insurance-mcp-client/src/client/prompts.py (SYSTEM_PROMPT + USER_TEMPLATE)
  - OpenAI chat completions (model: gpt-4.1-mini)
- The prompt includes:
  - the user question + state code
  - the SOURCES block
  - citation rules and a fixed output format
- Extra safety when citations are required:
  - Builds an explicit allow-list of citation tags derived from retrieved matches.
  - Instructs the model to copy citations only from that allow-list.

Node 6: citation_verify (defensibility gate)
- Function: verify_citations_node()
- Only enforced when require_citations is True.
- It verifies two things:
  1) citations in the answer match the retrieved match keys (file/doc_type/page/chunk)
  2) required sections have bullets that end with citations:
     - "Coverage answer"
     - "Key conditions / exclusions to watch" (minor header variations allowed)

Formatting normalization (before verification):
- Collapses hard-wrapped bullets in required sections (so one bullet = one line).
- Moves citations to the end of required bullets when the model inserts them mid-sentence.

One retry:
- If verification fails and we have not retried yet, it asks the model once more to fix formatting and use only allowed citation tags.
- If verification still fails, the run is blocked and the answer is suppressed (answer cleared).


5) MCP tool contract
--------------------

Server implementation:
- home-insurance-mcp/src/server/mcp_server.py
  The server uses a single dispatcher: _call_tool_impl(name, arguments)

All tools return JSON encoded as MCP TextContent.
On error, call_tool returns:
  {"status": "error", "tool": "<name>", "error": "<redacted error text>"}

Core tools:
- health
  Returns basic server status + docs_root.

- index_status
  Returns readiness signals:
  - Qdrant connectivity
  - collection existence and point count
  - OpenAI configured + optional ping result (cached)
  - OCR readiness (fitz + tesseract availability)
  - docs root path

- ingest_folder
  Returns summaries per file:
  - file_name, doc_type, text_chars, chunks_count
  Does not embed or write to Qdrant.

- index_folder_qdrant
  Indexes supported files under the folder:
  - PDF and common image types
  - chunks + embeddings + upsert to Qdrant

- retrieve_clauses
  Semantic search over indexed chunks:
  Returns:
  - score
  - file_name, doc_type, page_number (if present), chunk_index
  - snippet (first ~800 chars of the stored chunk text)

Job tools:
- start_ingest_job / start_index_job
  Returns {job_id}. Server runs work in a background thread.

- job_status
  Returns the job record with fields:
  - status/message
  - progress (files_total/files_done, chunk estimates, batches done, etc.)
  - result or error

Handoff tools:
- create_handoff_ticket
  Takes question/state/answer/sources and optional redacted match summaries.
  Stores in memory and appends to `.demo_state/handoff_tickets.jsonl` when local persistence succeeds.

- list_handoff_tickets
  Returns recent stored tickets.

Utility tool:
- normalize_quote_snapshot
  Normalization of pasted quote/rating text into key fields.

8) Extension points (where to add features safely)
--------------------------------------------------

Add a new server tool:
- home-insurance-mcp/src/server/mcp_server.py
  - Add a new `if name == "<tool_name>":` block in _call_tool_impl
  - Add a matching Tool schema in list_tools()
  - Keep outputs JSON and errors redacted

Add a new workflow step:
- home-insurance-mcp-client/src/client/graph.py
  - Add a node function
  - Wire it in build_graph() with explicit edges
  - Log a trace event so the UI audit panel stays useful

Change answer format or citation rules:
- home-insurance-mcp-client/src/client/prompts.py (format rules)
- home-insurance-mcp-client/src/client/graph.py (verification logic)
  Keep these two in sync. The whole "defensible output" behavior depends on it.
